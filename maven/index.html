    1. 基本概念

    1.1. 简介

    基于 Java 平台的自动化构建工具。
    构建：以java代码、html+css+js页面、图片等静态资源变成可发布项目的过程。
    构建工具发展历史：make -> ant -> maven -> gradle

    1.2. 作用

    • 将项目拆分成若干个模块。
    • 管理 jar 包(增加第三方 jar 包；管理 jar 包之间的依赖关系，自动关联下载所有依赖的 jar 包且不会冲突)。
      例如，引入commons-fileupload依赖会自动引入commons-io依赖。

    1.3. 项目构建重要环节

    • 清理：删除之前的编译结果，为重新编译做准备。
    • 编译：将 java 源文件编译成字节码文件。
    • 测试：针对项目中关键点进行测试。
    • 报告：记录展示测试结果。
    • 打包：将项目中包含的多个文件压缩成一个文件用于安装，将 java 项目打成 jar 包，将 web 项目打成 war 包。
    • 安装：将包放到本地仓库，供其他项目使用。
    • 部署：部署服务器准备运行，将 java、js、jsp 等文件进行筛选组装，变成一个可以直接运行的项目。

    2. 安装配置

    2.1. windows 环境

    1. 下载解压安装包。
      gz是mac版，zip是windows版本。

    2. 配置环境变量。

      # 配置JAVA_HOME（基于Java平台）。
# 配置MAVEN_HOME或者M2_HOME（解压后maven路径）。
# 配置PATH（解压后maven中bin的路径，%MAVEN_HOME%\bin）。

    3. 测试。

      mvn -v # 查看maven版本号

    4. 配置本地仓库。

      conf/settings.xml
      <!-- 默认本地仓库在.m2\repository -->
<localRepository>D:/mvnrep</localRepository>

    3. 核心概念

    3.1. 目录结构

    - 项目
    - src
        - main
            - java：程序代码
            - resources：资源代码，配置代码
        - test
            - java：测试代码
            - resources
    - pom.xml：项目对象模型，maven工程核心配置
    - target：编译后结果

    3.2. 常用命令

    # 执行命令必须在pom.xml所在目录执行
mvn clean # 清理，删除编译文件的目录target
mvn compile # 编译，第一次编译会比较慢，因为需要从中央仓库下载依赖
mvn test # 测试
mvn test-compile # 编译测试
mvn package # 打包，测试不通过无法打包
mvn install # 安装，将开发的模块安装到本地仓库中，供其他模块使用，位置由gav决定
mvn site # 生成站点

# 将指定jar包安装到本地仓库！！！
mvn install:install-file -Dfile=H:mybatis-ehcache-1.0.3.jar -DgroupId=org.mybatis.caches -DartifactId=mybatis-ehcache -Dversion=1.0.3 -Dpackaging=jar

    注意：maven的核心程序仅仅定义抽象的生命周期，具体的工作必须由特定的插件来完成，插件本身并不包含在maven的核心程序中。执行命令时，用到某个插件，会从本地仓库中查找...

    3.3. GAV 坐标

    GAV 坐标在本地仓库中唯一标识一个模块。

    • groupId：域名反转+项目名称。
    • artifactId：当前项目模块名称。
    • version：版本号。
      SNAPSHOT 快照，开发使用；RELEASE 发行版。

    3.4. 仓库

    • 本地仓库：
    • 远程仓库：
      私服，搭建在局域网环境，通过 nexus 搭建。
      中央仓库。
      中央仓库镜像，分担中央仓库的流量，提升用户访问速度。

    3.5. 生命周期

    Maven 生命周期定义各个构建环节的执行顺序。运行任何一个阶段的时候，它前面的所有阶段都会被运行。

    • clean 阶段：pre-clean，clean，post-clean。
    • default 阶段：compile（编译源代码），test-compile（编译测试源代码），test，package，install。
    • site 阶段：pre-site，site，post-site，site-deploy 部署。

    4. 依赖

    4.1. 作用

    使用某个存在的 jar 包或模块，可以采用依赖实现（去本地仓库、中央仓库中寻找）。

    4.2. 依赖范围与有效性

    • compile：编译有效，测试有效，部署有效。默认依赖范围。
    • test：编译无效，测试有效，部署无效。常见 junit.jar。
    • provided：编译有效，测试有效，部署无效。常见 servlet-api.jar，由于 tomcat 自带，如果设置 compile，可能版本冲突报错。

    4.3. 依赖排除

    <dependency>
    <groupId>com.fengbiao1</groupId>
    <artifactId>mango-schedule</artifactId>
    <version>0.0.1-SNAPSHOT</version>
    <!-- 要排除的依赖 -->
    <exclusions>
        <exclusion>
            <groupId></groupId>
            <artifactId></artifactId>
        </exclusion>
    </exclusions>
</dependency>

    4.4. 依赖传递性

    A 依赖 B，B 依赖 C，当且仅当 B 依赖于 C 的范围是 compile 时，A 依赖于 C。

    4.5. 依赖原则

    • 路径最短优先原则。
    • 路径长度相同：
      • 如果同一个 pom 文件中有两个相同的依赖（禁止），后面声明的依赖覆盖前面声明的依赖。
      • 如果在不同的 pom 文件中有两个相同的依赖，则先声明的依赖会覆盖后声明的依赖。

    5. 继承

    5.1. 作用

    在父工程中统一管理依赖信息。

    5.2. 继承 VS 依赖传递

    • 依赖传递：A 依赖于 B，B 依赖于 C，当且仅当 B 依赖于 C 在 compile 范围，则 A 依赖于 C。
    • 继承：A 继承于 B，则 A 可以使用 B 所有依赖。

    5.3. 示例

    1. 建立父工程，打包方式为 pom。
      java工程打包方式为jar，web项目打包方式为war，父工程、聚合打包方式为pom。
    2. 在标签 dependencyManagement 中管理依赖信息。
    3. 子工程继承父工程（如果子工程和父工程的 gv 一致，可以省略）。
    4. 在子工程声明需要使用的父工程依赖（如果版本号和依赖有效性与父工程一致，可以省略）。

    6. 聚合

    6.1. 作用

    将多个工程拆分为模块以后，需要手动逐个安装到仓库中，依赖才会生效，且如果修改源码后，也需要逐个手动进行 clean 安装。采用聚合批量进行 Maven 前置工程的安装。

    6.2. 聚合 VS 继承

    • 聚合：将大工程拆分成若干个子工程（子模块），聚合可以将拆分的多个子工程合起来。
    • 继承：通过父工程统一管理依赖版本。

    6.3. 示例

    在聚合工程中通过标签 modules 和 module 指定模块工程的相对路径。

    6.4. 注意事项

    1. module 顺序无所谓，maven 会自动识别。
    2. 聚合工程打包方式为 pom。
    3. 聚合子工程并不需要继承同一个父工程。

    7. 插件

    7.1. springboot 项目配置打包插件

    1. 如果以继承的方式引入 spring-boot-starter-parent。

      <build>
    <plugins>
        <plugin>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-maven-plugin</artifactId>
        </plugin>
    </plugins>
</build>

    2. 如果以依赖的方式引入 spring-boot-starter-parent，详见 spring-boot-starter-parent 对应 pom 文件。

      <build>
    <plugins>
        <plugin>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-maven-plugin</artifactId>
            <executions>
                <execution>
                    <id>repackage</id>
                    <goals>
                        <goal>repackage</goal>
                    </goals>
                </execution>
            </executions>
            <configuration>
                <mainClass>com.fengbiao1.mango.admin.MangoAdminApplication</mainClass>
            </configuration>
        </plugin>
    </plugins>
</build>

    7.2. 常用插件

    • spring-boot-maven-plugin：springboot 项目所必须，否则 jar 包无法运行，但是引入这个插件必须要有主启动类。
    • maven-compiler-plugin：可以定制化 jar 包。

    8. 其他

    8.1. 统一编码

    <properties>
    <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
</properties>

    8.2. 统一项目 jdk 版本

    • 右键项目 Build Path，选择 config，选择 library，删除旧的版本，新增新的版本 Add library（JRE System Liberary）。
    • 右键项目 properties，选择 Project Facets，选择 java 版本号。
    • 通过 maven 配置 jdk 版本。

    8.3. 配置多环境 profiles

    1. 配置 pom 文件。

      <profiles>
    <profile>
        <id>test</id> <!-- 唯一ID -->
        <properties>
            <profiles.active>test</profiles.active> <!-- 标签为key，将来使用；值为对应的profile值 -->
        </properties>
    </profile>
    <profile>
        <id>uat</id>
        <properties>
            <profiles.active>uat</profiles.active>
        </properties>
    </profile>
</profiles>

    2. application.yml 使用@profiles.active@。
